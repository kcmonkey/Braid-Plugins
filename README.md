## What is Braid?

[Braid](https://github.com/kcmonkey/Braid) is a VS Code extension (with standalone/Electron/Chrome targets)
that turns coding-agent conversations into a **node-based canvas (DAG)**. Each conversation round is a
**Board**; boards connect via context-inheritance edges; you can fork, merge (deduped combined context),
collapse history, and switch between Claude, Codex, DeepSeek, OpenRouter, and OpenAI-compatible providers on
the same canvas.

Plugins extend Braid by contributing **board elements, context providers, agent tools, board forms, board
pins, board actions, workspace panels, and more** — all through a provider-neutral plugin contract that
works with every engine.

## Plugin Catalog

| Plugin | ID | Default | Description |
|---|---|---|---|
| **Coordinator** | `coordinator` | on | Coordinates shared workspace resources across boards — agents negotiate editor sessions, builds, and file claims without stepping on each other. |
| **Knowledge Vault** | `knowledge` | on | Durable project knowledge vault under `.braid/knowledge/`. A dormant context provider tells the agent to record reusable facts as notes and read them on demand. |
| **Plan** | `plan` | off | Bind a board to a `.braid/plans/<name>/` plan. Shows current-phase status as a read-only chip, injects plan-context routing into the board's turns, and tracks phase completion. |
| **Tags** | `tags` | on | Derive and render board topic tags from the board's content using cheap-model classification against a curated tag vocabulary. |
| **Image Board** | `codex-image` | on | Generate image artifacts from a dedicated image board and reuse them as visual input on chat boards via typed board pins. |
| **Prompt Macros** | `prompt-macros` | on | Add configurable prompt buttons to settled boards. Define reusable prompt templates that inject into the composer with one click. |
| **Context Ref** | `context-ref` | off | Reference context-provider plugin. When enabled, injects a marker block into every turn. Used as an e2e test gate for the `contextProviders` plugin seam — **not a real feature**. |

### Coordinator

Coordinates shared workspace resources (editor sessions, build commands, file claims) across Braid boards
running concurrently in the same project. Agents running in different boards can negotiate resource access
through the `coordinate` agent tool — claim, release, wait, and request resources — and the host enforces
conflict prevention (e.g. two boards can't launch the same editor simultaneously). Ships with a workspace
panel, live coordination notices injected into board context, and a badge showing active claims.

### Knowledge Vault

A dormant context provider that teaches every board the **record / recall** protocol on `.braid/knowledge/`:

- **Record**: The agent writes durable, reusable, project-specific facts to `.braid/knowledge/<topic>.md`
  notes (title, claim, scope, evidence, metadata) and appends a routing entry to `_index.md`.
- **Recall**: Before answering from memory, the agent reads `_index.md` first, then reads the specific
  note(s) relevant to the question. Note bodies are never auto-injected — only bounded index metadata
  enters context, keeping token usage low even with hundreds of notes.

Notes carry a typed lifecycle (`current` / `stale` / `superseded` / `disputed`) so outdated knowledge is
preserved but excluded from default routing.

### Plan

Binds a board to a `.braid/plans/<PlanName>/` plan folder using Braid's standard contract format
(`contract.md` + `current-phase.md` + `decisions.md` + `history.md` + `evidence/`). The plugin:

- Renders a compact inline chip (`◆ planId · phase · ✓n/n`) on every bound board, including far-far
  collapsed views.
- Injects plan-aware context routing: agents are directed to plan files for plan-related intent and kept
  out of plan execution for unrelated tactical work.
- Exposes a `runPolicies` contribution so autonomous plan runs know to continue through all phases.
- Plan binding survives compact checkpoints and fork/merge via the generic plugin inheritance seam.

### Tags

Derives up to 3 topic tags per board from a curated, provider-neutral tag vocabulary (14 tags: plan, debug,
refactor, review, feature, fix, docs, test, commit, build, deploy, config, deps, research). Tags are
generated by cheap-model (Haiku-equivalent) classification after each board settles, persisted on the board,
and rendered as colored chips on both far and detail card views. Clearing on follow-up / fuse ensures tags
stay current with the board's actual content.

### Image Board

Contributes a **board form** (`codex-image`) that creates a dedicated image-generation board with its own
composer UI. Generated images are rendered as thumbnails (far/detail views). The plugin also contributes
typed **board pins** (`image` data type): image boards expose image output pins, chat boards expose image
input pins — connect them to feed generated images as visual context into a chat turn. Supports regenerate
and multiple incoming image connections.

### Prompt Macros

Adds configurable **board actions**: prompt-buttons that appear on settled boards. Each macro is a named
prompt template. Clicking a macro button injects the template text into the composer — useful for recurring
workflows like "review this for correctness bugs", "summarize the decisions in this branch", or
"check for security issues".

### Context Ref

A **reference implementation** of the `contextProviders` plugin seam. When enabled, it injects a
`[ContextRef]` marker block into every turn's system context. Not intended for production use — it exists
to exercise and gate the context-provider plugin contract end-to-end. Disabled by default.

## Installation

### From the Braid Plugin Settings

1. Open Braid's **Plugin Settings** panel (⚙ → Plugins → Manage).
2. Each official plugin appears with its manifest metadata (id, version, description, enable/disable toggle).
3. Enable the plugins you want. They take effect immediately — no restart needed.

### Manual / Development

Official plugins ship as **builtins** inside Braid's `plugins/builtin/` directory and are discovered at
startup from the project's `.braid/plugins.json` or user-level plugin manifest. External/user plugins
follow the same manifest format and are loaded from `.braid/Plugins/`.

## Plugin Manifest Format

Every plugin declares a `plugin.json` at its root:

```json
{
  "id": "my-plugin",
  "name": "My Plugin",
  "version": "1.0.0",
  "description": "What it does.",
  "defaultEnabled": true,
  "canDisable": true,
  "scope": "builtin",
  "runtime": "builtin",
  "contributes": {
    "boardElements":   ["my-plugin"],
    "boardForms":      ["my-plugin"],
    "boardActions":    ["my-plugin"],
    "boardPins":       ["my-plugin"],
    "contextProviders": ["my-plugin"],
    "agentTools":      ["my-tool"],
    "toolMiddleware":  ["my-middleware"],
    "hostProjectState": ["my-state"],
    "hostServices":    ["my-service"],
    "turnContext":     ["my-context"],
    "liveMessages":    ["my-messages"],
    "workspacePanels": ["my-panel"],
    "workspaceBadges": ["my-badge"],
    "webviewMessages": ["my-webview-msgs"],
    "runPolicies":     ["my-policy"]
  }
}
```

| Contribution | Type | Description |
|---|---|---|
| `boardElements` | webview | Render a read-only element inside each board card (e.g. tags, plan chip). |
| `boardForms` | webview | Register a custom board creation form with its own composer UI. |
| `boardActions` | webview | Add action buttons to settled boards (e.g. prompt macros). |
| `boardPins` | webview | Contribute typed input/output pins for cross-board data flow. |
| `contextProviders` | webview | Inject text into every turn's system context. |
| `agentTools` | host | Register an agent-callable tool (in-process MCP / dynamic tools). |
| `toolMiddleware` | host | Intercept and observe tool calls before execution. |
| `hostProjectState` | host | Provide project-scoped key-value state readable by all boards. |
| `hostServices` | host | Register a long-lived host service with lifecycle hooks. |
| `turnContext` | host | Inject context text into each turn. |
| `liveMessages` | host | Push live messages into streaming board turns. |
| `workspacePanels` | webview | Render a panel in the workspace sidebar. |
| `workspaceBadges` | webview | Render a status badge in the workspace chrome. |
| `webviewMessages` | both | Register custom protocol message handlers. |
| `runPolicies` | host | Declare a named run policy for autonomous plan execution. |

## Developing a Plugin

1. **Create a directory** with your plugin id as the name.
2. **Add a `plugin.json`** manifest declaring `id`, `name`, `version`, `description`, and `contributes`.
3. **Implement the contributed slots** as exported functions/components matching the plugin API contracts.
4. **Add tests** — plugins can import their own pure helpers and test with vitest. Shared test utilities
   live in the `shared/` directory.
5. **Run the boundary test**: `npx vitest run plugin-boundary.test.ts`. This enforces that plugins never
   import core Braid internals (webview merge, coordination model, extension host, VS Code API).

### Boundary Rules

Plugins are **isolated from Braid core**. The boundary test enforces:

- **No imports** from `src/webview/merge`, `src/coordination`, `src/plugin-api`, `src/plugin-runtime`,
  `src/webview/main`, `src/extension`, `src/app`, or `src/host`.
- **No references** to `vscode`, `acquireVsCodeApi`, `window.BraidPluginRuntime`, or `BraidCore`.

Plugins communicate with Braid exclusively through the **plugin contribution contracts** declared in their
manifest. This isolation is what makes plugins portable across Braid's VS Code, standalone, Electron, and
Chrome targets.

## Testing

```bash
# Run plugin boundary checks
npx vitest run plugin-boundary.test.ts

# Run all plugin tests
npx vitest run

# Run tests for a specific plugin
npx vitest run coordinator/coordinator.test.ts
npx vitest run knowledge/parse.test.ts
npx vitest run plan/detect.test.ts
```
